1. 从并行编程模型说起

假设我们有一段最基础的标量顺序代码，对一个数组做逐元素乘法：

for (i = 0; i < N; i++)
    A[i] = B[i] * C[i];

这段循环天然具有数据级并行（DLP）：每次迭代之间相互独立，没有数据依赖。要 exploit 这种并行性，历史上出现过三种典型编程模型。

SISD（单指令单数据）：完全顺序执行。现代 CPU 会通过流水线、超标量发射、乱序执行等方式在硬件层面动态展开循环，把不同迭代塞进指令窗口同时执行。但这本质上仍是对顺序代码的加速，程序员无需关心并行。

SIMD（单指令多数据）：程序员或编译器显式生成向量指令，一次性对多个数据元素执行相同操作。例如向量加载 VLD、向量乘加 VADD。这种模型非常适合向量处理器和带有 AVX/SSE 单元的 CPU。

SPMD / MIMD（单程序多数据）：把每一次迭代都变成独立的线程，所有线程执行同一段程序，但操作不同的数据。这是 GPU 编程的核心思想。从程序员视角看是 SPMD，从硬件架构视角看则是 SIMT（Single Instruction Multiple Thread）。

进入 21 世纪，机器视觉、语音识别、深度学习等应用的爆发，使得大规模数值计算成为常态。这些计算往往天然具备数据并行特征，SIMD 类架构（包括向量 SIMD、子字 SIMD 和 SIMT/GPU）成为最高效的执行载体。

2. GPU 与 CPU：两种截然不同的设计哲学

理解 GPU 架构的第一步，是认识到它和 CPU 根本不是为同一目标设计的。

CPU：面向延迟（Latency-Oriented）

  • 高主频（~3.0 GHz），追求单线程最低响应时间。
  • 大容量缓存（L1 ~KB 级，L2/L3 ~MB 级），用缓存命中把长延迟内存访问转化为短延迟访问。
  • 复杂的控制逻辑：分支预测、乱序执行（OOO）、数据转发、超标量发射。
  • 强大的 ALU，降低单条指令的执行延迟。

GPU：面向吞吐量（Throughput-Oriented）

  • 中等主频（~1 GHz），不追求单线程快，而追求单位时间内完成的工作总量。
  • 极小的缓存（Scratchpad / Shared Memory），目的不是降低延迟，而是提升内存带宽利用率。
  • 极简的控制逻辑：无分支预测、无数据转发，省去大量晶体管。
  • 海量 ALU，数量可达数千个，每个 ALU 流水线极深，长延迟但高吞吐。
  • 不隐藏延迟，而是用超量线程（massive multithreading）来容忍延迟：当一个 warp 因访存而阻塞时，硬件立刻切换去执行另一个就绪 warp。

结果是：CPU 在串行代码上可能比 GPU 快 10 倍，而 GPU 在高度并行的代码上可能比 CPU 快 10 倍甚至更多。这就引出了异构计算的范式——CPU 负责串行部分和调度，GPU 负责并行计算密集型部分。

3. CUDA 编程模型：Grid → Block → Thread

NVIDIA 在 2006 年推出 GeForce 8800 时统一了 GPU 的所有并行形式为 CUDA Thread，并提供了一套类 C 语言（CUDA C）来编程。其编程模型围绕三个核心抽象展开：

  1. 线程组层次结构（Hierarchy of Thread Groups）
  2. 共享内存（Shared Memories）
  3. 屏障同步（Barrier Synchronization）

CUDA 的执行组织呈现三级层次：

  • Kernel（核函数）：在 GPU 上并行执行的函数，由所有线程共同运行。
  • Grid（网格）：一次内核调用的所有线程的总体集合。
  • Block（线程块）：Grid 被划分为多个 Block，Block 内线程可以协作。
  • Thread（线程）：最基本的执行单元，每个线程有独立的寄存器状态和线程 ID。

调用 Kernel 时使用特有的执行配置语法：

__global__ void vecAdd(float* A, float* B, float* C, int n) {
    int i = blockIdx.x * blockDim.x + threadIdx.x;
    if (i < n) C[i] = A[i] + B[i];
}

// 主机代码
int threadsPerBlock = 256;
int blocksPerGrid = (N + threadsPerBlock - 1) / threadsPerBlock;
vecAdd<<>>(d_A, d_B, d_C, N);

这里 <<>> 就是指定 Grid 和 Block 的组织方式。每个线程通过内置变量 blockIdx、threadIdx、blockDim、gridDim 计算自己在全局数据中的位置，从而把同一段代码映射到不同数据上。

这种组织方式的优势在于可扩展性：算法写好之后，无论 GPU 上有多少个 SM（Streaming Multiprocessor），硬件都可以自动把 Block 分发到空闲的 SM 上执行，无需重写代码。

4. GPU 内存层级与访存优化

GPU 的内存系统是分层的，每一级在容量、延迟和可见性上都不相同。写 CUDA 代码时，绝大部分性能调优工作都在围绕"数据放在哪、怎么搬、怎么访"展开。

内存类型位置延迟可见范围生命周期
Register片上~1 cycle单个 ThreadThread
Local Memory片外（Device Memory）高单个 ThreadThread
Shared Memory片上（Scratchpad）~20-30 cyclesBlock 内所有 ThreadBlock
Global Memory片外（DRAM）~400-800 cycles所有 Thread + Host应用级
Constant / Texture片外（只读缓存）中所有 Thread应用级

Global Memory：容量最大（几 GB 到几十 GB），但延迟最高。一个 warp（32 个线程）同时访问 Global Memory 时，硬件会尝试把相邻线程的访存请求合并（Coalesce）成一次或几次缓存行宽度的物理事务。如果线程访问的地址连续且对齐，就能达到接近峰值带宽；如果访问模式随机（Strided / Scattered），则会产生大量独立事务，带宽骤降。

Shared Memory：位于 SM 片上，可被 Block 内线程共享，延迟远低于 Global Memory。它的本质是一块Scratchpad Memory（软件管理的暂存器），而非硬件自动管理的 Cache。程序员需要显式地把数据从 Global Memory 加载到 Shared Memory，算完再写回。使用 Shared Memory 的两个主要目的是：

  1. 复用数据：一个数据元素被 Block 内多个线程使用时，只需从 Global Memory 读一次。
  2. 避免非合并访存：通过 Shared Memory 做数据重排（tiling），把不规则的全局访存转化为规则的片上访问。

Register：每个线程私有的片上存储，数量极多但有限（通常每个 SM 有 64K-256K 个 32-bit 寄存器）。如果寄存器溢出，编译器会把变量 spill 到 Local Memory，性能会严重下降。

5. Warp 调度与分支发散

GPU 的实际调度单位不是单个 Thread，而是 Warp（NVIDIA 术语，AMD 称 Wavefront，大小通常为 32 或 64 个线程）。一个 Warp 内的所有线程共享同一个程序计数器（PC），以 lock-step 方式执行同一条指令，这就是 SIMT 的核心。

Warp 调度器每个周期从就绪的 warp 中挑选一个发射指令。由于 GPU 上通常同时存在数十个 warp，当某个 warp 因为等待内存数据而阻塞时，调度器可以零开销切换到另一个就绪 warp，用线程级并行（TLP）隐藏内存延迟。这种机制被称为延迟隐藏（Latency Hiding）。

但 SIMT 有一个致命的弱点：分支发散（Branch Divergence）。当 Warp 内的线程遇到条件分支（如 if-else）且不同线程走不同路径时，硬件无法让一部分线程执行 if 分支而另一部分空闲。相反，硬件会串行化执行两条路径：先执行所有走 if 路径的线程（走 else 的线程被掩码掉），再执行所有走 else 路径的线程。这导致 warp 的有效利用率下降，性能减半甚至更低。

避免分支发散的常见策略：

  • 让分支条件以 Warp 为单位对齐（例如按 warp 大小对齐数据索引）。
  • 使用 warp shuffle 指令（__shfl_sync）在 warp 内直接交换数据，避免 Shared Memory 往返。
  • 用 predicates 或位掩码手动控制线程活跃性。
6. CPU-GPU 异构编程范式

现代 GPU 并非独立运行，而是作为 CPU 的协处理器存在。典型的异构系统架构中，CPU（Host）和 GPU（Device）拥有各自独立的内存空间，通过 PCIe 总线或片上互联（如 NVLink、Infinity Fabric）通信。

{{< docpage "pdf/computer-system/L03c-GPGPU-Arch.pdf" page=11 >}}

一个完整的 CUDA 程序执行流程如下：

  1. 分配设备内存：在 GPU 显存中分配空间（cudaMalloc）。
  2. 数据传输：把输入数据从主机内存复制到设备内存（cudaMemcpy Host→Device）。
  3. 启动 Kernel：配置 Grid/Block 参数，在 GPU 上执行并行计算。
  4. 结果回传：把计算结果从设备内存复制回主机内存（cudaMemcpy Device→Host）。
  5. 释放资源：释放设备内存（cudaFree）。

PCIe 的带宽（~32 GB/s for PCIe 4.0 x16）远低于 GPU 显存带宽（~1-2 TB/s），因此数据传输开销往往是性能瓶颈。优化策略包括：

  • Zero-Copy / Unified Memory：让 CPU 和 GPU 共享同一虚拟地址空间，由硬件自动处理页迁移。
  • 数据驻留 GPU：把多轮计算串成流水线，中间结果不搬回主机。
  • 异步传输与计算重叠：使用 CUDA Stream 让数据传输和 Kernel 执行并行。
7. 案例：矩阵乘法的 CUDA 实现思路

矩阵乘法 C = A × B 是理解 GPU 并行编程的经典案例。一个朴素的实现让每个线程计算 C 中的一个元素：

__global__ void matMul(float* A, float* B, float* C, int N) {
    int row = blockIdx.y * blockDim.y + threadIdx.y;
    int col = blockIdx.x * blockDim.x + threadIdx.x;
    float sum = 0.0f;
    for (int k = 0; k < N; k++)
        sum += A[row * N + k] * B[k * N + col];
    C[row * N + col] = sum;
}

这个实现的问题在于：计算 C 中同一行的不同线程，会重复读取 A 的同一行数据；计算 C 中同一列的不同线程，会重复读取 B 的同一列数据。而且 A 按行访问是连续的（合并访存），但 B 按列访问的步长是 N（非合并访存），带宽利用率极低。

Tiled（分块）优化是标准解法：

  1. 把 A 和 B 都划分为大小为 TILE_SIZE × TILE_SIZE 的小块。
  2. 一个 Block 负责计算 C 中的一个 tile。
  3. Block 内所有线程先把当前步对应的 A-tile 和 B-tile 从 Global Memory 协作加载到 Shared Memory。
  4. 在 Shared Memory 上做局部点积计算。
  5. 沿 K 维度滑动，重复加载和计算，最后把累加结果写回 Global Memory。

通过 Tiling，原本对 Global Memory 的 O(N³) 次非合并访问，被转化为 O(N³ / TILE_SIZE) 次合并访问加上大量高速 Shared Memory 复用。实际中配合 register 展开和 warp 级优化，可以获得接近峰值算力的性能。
