核心问题：LIC 够鲁棒吗？谁来测、怎么测、扰动打在哪一层？
问题定义
LIC 鲁棒性：在哪里、抵抗什么？

LIC（Learned Image Compression，端到端学习的神经图像压缩）在压缩效率上已超越传统编码器，但其比特流极度脆弱。当传输链路出现异常时——丢包，或注入扰动——解码重建质量会急剧下降。

「LIC 鲁棒吗」看似简单，实则需要先厘清两个前提：

前提一：扰动发生在哪里？

传输链路有三个位置可以发生异常：

  • 信道层：bitstream 在传输中被丢弃（packet loss / bit erasure）
  • 编码器前：source image 被注入对抗扰动（adversarial perturbation）
  • 解码器侧：decoder 本身被攻击

不同位置的扰动激活的防御机制完全不同。

前提二：「抵抗」是什么意思？

「鲁棒」可以指向两种能力：

  • 韧性（Resilience）：扰动发生后，系统能恢复或维持可用输出
  • 免疫（Robustness）：扰动无法显著影响输出质量

这两种能力并不等价——一个系统可以有韧性但无免疫（有恢复机制但质量下降明显），也可以有免疫但无韧性（质量不受影响但无法处理实际丢包）。

核心问题：LIC 面临两条独立的威胁模型——擦除信道（信息缺损）与篡改信道（信息污染）——它们要求的防御机制完全不同，且没有统一的评测框架。

这根线从两条独立的实验脉络展开。第一条脉络（B1）把 bitstream 丢掉一部分，看 decoder 能不能在不完整的 latent 上重建出可用的图。第二条脉络（B2）不动 bitstream，在 source image 进入编码器之前注入对抗扰动，看整条编码-解码链能不能扛住一个肉眼看不见的修改。两条脉络各自产出了有效的设计回应和脆弱性证据，但它们面对的不是同一种威胁。

威胁模型分裂
B1 擦除信道 · B2 篡改信道

B1 和 B2 表面上都在回答"LIC 鲁棒吗"，但它们的威胁模型在信道层面上就不在同一坐标系。

B1 — 擦除信道

bitstream 在传输中被丢了一部分，decoder 拿到的 latent representation 是不完整的——少了什么。ResiComp 和 Satellite 都是在这个前提下的设计回应：一个接受 AR context 依赖但训练韧性来兜底，另一个从结构上砍掉依赖来断传播路径。它们的 failure ceiling 是可比较的，因为攻击者能做的事情只有一件：让更多的 bit 丢掉。

B2 — 篡改信道

bitstream 完整到达，但 source image 在进入编码器之前就被动了手脚——decoder 拿到的是被改了什么。Wu2024 证明的脆弱性不是"丢包后重建崩溃"，而是"一个肉眼看不见的扰动，经过编码器内部被放大到整张图不可用"。RobustNIC 的 defense 是朴素的对抗训练——让模型见过攻击就不那么怕，但没有堵住传播路径本身。

结论：B1 和 B2 各自有效，但架构要求方向根本不同——B1 解决冗余和隔离问题，B2 解决对抗鲁棒性问题。两个方向没有互补关系，整合的路径本身就是错的。
Gap 与 Thesis
结构性 Gap 与未正确提出的问题
根本问题：B1 和 B2 面对的也许根本就不是同一种鲁棒性，强行找统一评测框架本身可能就是一个错误的问题。
