Part 1
为什么需要双量化器？
统一图像 tokenizer 面临着本质性的矛盾：理解任务需要高层语义特征（"这张图片里有什么"），生成任务需要低层像素特征（"这张图片长什么样"）。现有的解决方案可以分为两条路线：
  • 联合训练路线（VILA-U, SDE, TokenFlow）：在单一码本内同时优化语义对齐损失和像素重建损失，但两个目标在梯度层面互相干扰
  • 解耦码本路线（SemHiTok）：先用纯语义码本做量化，再为每个语义 code 构建独立的 pixel sub-codebook，实现层级分解
ComTok 的思路介于两者之间——不追求完全解耦，而是引入一个可调节的比率 v，让模型能够在这两种极端之间连续插值。
核心问题：能不能找到一个统一 tokenizer，用一个参数 v 控制"多少比例的特征给像素、多少给语义"，从而根据任务需求自由调节？
Part 2
模型架构
ComTok 的核心是一个双量化器结构，流程如下：
  1. 输入图像经过 Patchify 得到 $N$ 个 patches，并额外添加 $M$ 个可学习的 semantic tokens
  2. 所有 $N + M$ 个 tokens 一起通过 共享 Encoder，利用自注意力让语义 token 与图像 token 相互交换信息
  3. Encoder 输出的 tokens 分为两路：
    • $N$ 个 image tokens → Image Quantizer（像素级码本）
    • $M$ 个 semantic tokens → Semantic Quantizer（语义级码本）
  4. 两组量化后的 tokens 在通道/序列维度拼接，送入 Decoder 重建图像
其中，图像 token 的比例 v 定义为： $$v = \frac{N_{\text{image}}}{N_{\text{image}} + N_{\text{semantic}}}$$ 当 $v = 0$ 时，所有 tokens 都通过 Semantic Quantizer，模型退化为纯语义 tokenizer；当 $v = 1$ 时，所有 tokens 都通过 Image Quantizer，模型退化为标准 VQGAN/MaskGIT。通过调节 $v$，ComTok 可以在语义理解和像素重建之间连续插值。
与 SemHiTok 的区别：SemHiTok 的 SGHC 是串行层级结构——先确定语义 code，再在其子空间内量化像素。ComTok 的量化是并行的——两组 token 各自量化后拼接，通过 Encoder 的共享自注意力建立联系。
Part 3
初步验证：v 对重建质量的影响
为了验证双量化器的可行性，我们首先进行了一组 v 参数扫描实验，固定模型结构，只改变图像 token 的比例。使用 LPIPS 作为重建质量的评估指标。
图 1：Image token 比例 v 对 LPIPS 的影响。随着 v 从 0 增大到 1，LPIPS 单调下降，重建质量持续提升。

关键发现

  1. 图像 token 越多，重建越好：v 从 0.00 → 0.25 → 0.50 → 1.00，LPIPS 从 0.3016 单调下降到 0.1926。这一趋势清晰且稳定，说明 image tokens 对重建的贡献是累加的。
  2. v = 1.00 超越 MaskGIT baseline：虽然 v = 1.00 相当于全图像 token 配置，但联合训练语义 token 的 Encoder 可能反而促进了图像特征的学习（通过共享自注意力的信息交换）。
  3. 全语义（v = 0.00）损失最大：纯语义 tokenizer 的 LPIPS = 0.3016，与 MaskGIT 的 0.1963 差距明显——这证实了语义 token 确实会丢失像素信息，双量化器的存在是必要的。
结论：双量化器结构是可行的——增加图像 token 比例 v 能单调提升重建质量。下一步需要验证：降低 v（即增加语义 token 比例）是否能在保持可接受重建质量的同时提升理解性能。
Part 4
后续工作与开放问题

待完成的实验

  • 理解任务评估：在 v=0.25/0.50 的中间配置下，评估多模态理解性能（MME-P, SEED-Bench, MMBench）。核心问题是：多大的 v 可以"免费"获得重建能力而不损害理解？
  • 与 SemHiTok / TokenFlow 的系统对比：在相同分辨率、相同 LLM backbone 下比较 rFID 和理解指标
  • v 的连续控制：如果 v 可以在推理时调节，ComTok 将成为一个"可伸缩"的统一 tokenizer——理解任务用低 v，生成任务用高 v

开放问题

  • 两个码本的容量分配：Image Quantizer 和 Semantic Quantizer 各自需要多少码字？码本容量是否应该随 v 动态调整？
  • 共享 Encoder vs 独立 Encoder：当前设计使用共享 Encoder，让两组 token 通过自注意力交换信息。如果换成独立 Encoder（像 Janus），结果会如何？
  • v 与任务的适配：是否可以在训练中引入一个可学习的门控，让模型自动决定每个 patch 走哪条路？
